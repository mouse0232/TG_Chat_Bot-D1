# 集成 AI 反骚扰功能 - 需求文档

## 1. 功能概述

在现有本地反骚扰检测的基础上，集成 AI（大语言模型）驱动的垃圾信息检测能力，实现对广告、诈骗、钓鱼等语义层面骚扰内容的智能识别与拦截。

## 2. 功能范围

### 2.1 本次实现范围

- LLM API 集成：调用 OpenAI 兼容接口进行垃圾信息语义检测
- AI 信任列表系统：自动识别可信用户，跳过 AI 检测以降低 API 成本
- 与本地检测并存：本地规则优先拦截，通过后再进入 AI 检测
- 管理员命令：手动信任/取消信任用户
- 管理面板：AI 反骚扰配置开关

### 2.2 未来扩展

- 多模型切换（不同 LLM 服务商）
- 自定义 prompt 模板
- 检测结果统计与分析

## 3. 核心设计原则

### 3.1 AI 信任列表与黑白名单完全解耦

AI 信任列表（`user_trust` 表）与项目的黑白名单（`users.is_blocked`）是两个独立的维度：

| 维度 | 存储位置 | 控制范围 | 说明 |
|------|----------|----------|------|
| 黑名单 | `users.is_blocked` | 消息是否放行 | 项目原有的封禁机制，控制消息能否通过 |
| AI 信任列表 | `user_trust.trust_status` | 是否跳过 AI 检测 | 仅控制是否需要调用 LLM API，不影响消息放行 |

**两者不发生任何关系**：
- 进入 AI 信任列表（trusted） ≠ 进入项目白名单（解除封禁）
- 被项目黑名单封禁 ≠ 从 AI 信任列表移除
- AI 信任列表用户触发本地规则被拉黑时，无需同步清理 AI 信任状态

**流程上的保证**：黑名单检查在流程前端（`handleVerifiedMsg` 第一步检查 `is_blocked`），黑名单拦截后消息不会到达 AI 检测环节，因此两者天然不冲突。

### 3.2 信任每日重置

AI 信任基于"当日连续行为"而非历史累计，每日重置：
- 用户当天连续通过 N 次 AI 检测 → 进入 AI 信任列表（trusted）
- 第二天首次发消息时 → 信任计数归零，状态回到 `new`，需重新当日积累
- 这样设计后，**拉黑/解封与 AI 信任完全无关**：不管用户是否被拉黑过，第二天都是重新计算

## 4. 详细需求

### 4.1 AI 垃圾信息检测

| 需求项 | 说明 |
|--------|------|
| 检测时机 | 本地反骚扰检测通过后，消息转发前 |
| 检测对象 | 仅非 AI 信任用户的文本消息 |
| 检测范围 | 广告营销、诈骗钓鱼、恶意链接、骚扰辱骂、刷屏等 |
| 检测方式 | 调用 LLM API，传入发送者姓名和消息内容 |
| 检测结果 | LLM 返回 "SPAM:原因" 或 "CLEAN" |
| 检测失败策略 | fail-open：AI 不可用时放行消息，记录错误日志 |

**检测流程**：

```
用户消息 → 本地反骚扰检测 → 通过 → 黑名单检查 → AI 信任检查
                                                       ↓
                                                  trusted → 直接转发（跳过 AI）
                                                  非trusted → AI 检测
                                                              ↓
                                                        SPAM → 拉黑+通知管理员
                                                        CLEAN → 当日信任计数+1 → 转发
```

### 4.2 AI 信任列表系统

| 需求项 | 说明 |
|--------|------|
| 信任状态 | new（新用户）、trusted（AI信任，免检）、monitoring（曾被判垃圾，重新积累中） |
| 自动加信条件 | 当日连续通过 N 次 AI 检测（无累计垃圾次数门槛） |
| 每日重置 | 第二天首次发消息时，consecutive_clean_count 归零，trust_status 回到 new |
| 信任用户行为 | 当日消息直接转发，跳过 AI 检测（第二天需重新积累） |
| 垃圾消息打断 | AI 检测为 SPAM 后：当日计数归零，状态转为 monitoring |
| monitoring 用户 | 从 1 重新累计当日通过次数，达到阈值后转为 trusted |
| 管理员手动信任 | `/trust` 命令直接加入 AI 信任列表 |
| 管理员取消信任 | `/untrust` 命令移出 AI 信任列表 |
| 与黑白名单关系 | 完全解耦，互不影响 |

**AI 信任状态流转**：

```
每日重置: new（所有用户第二天回到 new）

new ──当日AI通过──→ counting(1) ──当日AI通过──→ counting(N) ──达到阈值──→ trusted（当日免检）
  │                                                    ↑
  │              /trust 命令 ────────────────────────────┘
  │
  └─AI检测SPAM──→ monitoring ──当日AI通过──→ re-counting ──达到阈值──→ trusted
                      ↑
                      │ /untrust 命令
                      │
               trusted ──────────────────────────────────┘

第二天: trusted/monitoring/new ──首次消息──→ new（每日重置）
```

### 4.3 管理员命令

| 命令 | 使用方式 | 动作 |
|------|----------|------|
| `/trust` | 在转发消息下回复 | 立即将用户加入 AI 信任列表（当日免检） |
| `/untrust` | 在转发消息下回复 | 将用户移出 AI 信任列表 |

### 4.4 配置项

| 配置键 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `enable_ai_anti_harassment` | boolean | `false` | AI 反骚扰总开关 |
| `ai_anti_harassment_trust_threshold` | number | `3` | 当日连续通过次数阈值 |
| `ai_anti_harassment_notify_auto_whitelist` | boolean | `true` | 自动加入 AI 信任列表时是否通知管理员 |

**环境变量**：

| 变量名 | 说明 | 示例 |
|--------|------|------|
| `LLM_API` | LLM API Base URL | `https://api.openai.com/v1` |
| `LLM_MODEL` | LLM 模型名称 | `gpt-4o-mini` |
| `LLM_KEY` | LLM API Key | `sk-...` |
| `LLM_TIMEOUT_MS` | API 调用超时（毫秒） | `5000` |

### 4.5 垃圾信息拦截动作

| 场景 | 动作 |
|------|------|
| AI 检测为 SPAM | 设置 `is_blocked=true`（进入项目黑名单），记录垃圾原因到 `user_info`，调用 `manageBlacklist()`，发送拦截提示 |
| 管理员收到通知 | 在管理群组转发垃圾消息 + AI 分析报告 |
| 用户收到提示 | "您的消息因包含垃圾信息已被过滤。如有疑问，请联系管理员。" |
| AI 信任列表影响 | AI 检测为 SPAM 时，AI 信任计数归零、状态转为 monitoring（与项目黑名单无关） |

## 5. 业务流程

### 5.1 完整消息处理流程

```
用户发送消息
    │
    ▼
┌───────────────────┐
│ 本地反骚扰检测      │
│ (checkUser/        │
│  checkMessage)     │
└─────────┬─────────┘
          │ 通过
          ▼
┌───────────────────┐     ┌───────────────────┐
│ 黑名单/限流检查    │────▶│ 触发 → 终止流程    │
│ (is_blocked等)    │     └───────────────────┘
└─────────┬─────────┘
          │ 通过
          ▼
┌───────────────────┐
│ AI 反骚扰总开关？   │
└─────────┬─────────┘
     ┌────┴────┐
     ▼         ▼
  关闭        开启
     │         │
     ▼         ▼
  正常流程   ┌───────────────────┐
             │ AI 信任检查          │
             │ (trust_status)     │
             │ + 每日重置          │
             └─────────┬─────────┘
                  ┌────┴────┐
                  ▼         ▼
              trusted     非trusted
              (当日免检)     │
                  │         ▼
                  ▼     ┌───────────────────┐
              直接转发   │ AI 垃圾信息检测     │
                        │ (LLM API)          │
                        └─────────┬─────────┘
                             ┌────┴────┐
                             ▼         ▼
                          CLEAN      SPAM
                             │         │
                             ▼         ▼
                        当日信任   项目黑名单拉黑
                        计数+1    +AI信任归零
                             │       +通知管理员
                        ┌────┴────┐
                        ▼         ▼
                    达到阈值     未达阈值
                        │         │
                        ▼         ▼
                    当日免检     正常转发
                    (trusted)
```

## 6. 非功能需求

### 6.1 性能要求

- AI API 调用超时上限 5 秒，超时后 fail-open
- AI 信任用户不受 AI 检测延迟影响
- 本地检测优先拦截，减少不必要的 AI API 调用

### 6.2 成本控制

- AI 信任列表系统大幅减少 AI API 调用量
- 仅文本消息触发 AI 检测（非文本类消息仅做本地检测）
- 仅非 AI 信任用户触发 AI 检测

### 6.3 可靠性

- AI API 不可用时 fail-open，不阻断正常消息
- AI 检测错误不影响本地反骚扰功能
- AI 信任数据持久化到 D1

### 6.4 兼容性

- 与本地反骚扰检测并存，本地优先
- 与项目黑白名单系统完全解耦，互不影响
- 与现有管理面板兼容
- Cloudflare Workers 环境兼容（使用 fetch API，不依赖 Node.js SDK）

## 7. 验收标准

- [ ] AI 检测为 SPAM 的消息被拦截，用户进入项目黑名单
- [ ] AI 检测为 CLEAN 的消息正常转发
- [ ] AI 信任用户当日消息直接转发，不调用 AI API
- [ ] 当日连续通过 N 次 AI 检测后进入 AI 信任列表
- [ ] 第二天首次发消息时信任计数归零，状态回到 new
- [ ] 发送垃圾消息后当日信任计数归零，状态转为 monitoring
- [ ] AI 信任用户触发本地规则被拉黑时，AI 信任状态不受影响（两者解耦）
- [ ] `/trust` 命令手动加入 AI 信任列表
- [ ] `/untrust` 命令移出 AI 信任列表
- [ ] AI API 不可用时消息正常放行（fail-open）
- [ ] AI API 调用超时时消息正常放行
- [ ] 可通过配置开启/关闭 AI 反骚扰功能
- [ ] 管理员收到垃圾消息分析报告
- [ ] 与本地反骚扰功能无冲突
- [ ] AI 信任列表与项目黑白名单互不影响（解耦验证）
