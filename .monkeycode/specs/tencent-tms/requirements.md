# 集成腾讯文本内容安全服务 - 需求文档

## 1. 功能概述

在现有本地反骚扰检测的基础上，集成腾讯云文本内容安全（TMS）服务，实现对广告、诈骗、色情、暴恐等骚扰内容的快速识别与拦截。TMS 服务与 AI 反骚扰功能互斥（只能二选一），但功能逻辑与 AI 反骚扰保持一致。

## 2. 功能范围

### 2.1 本次实现范围

- 腾讯 TMS API 集成：调用 TextModeration 接口进行文本内容安全检测
- 信任列表系统：复用 AI 反骚扰的 user_trust 表和信任机制（每日重置、连续通过晋升）
- 与本地检测并存：本地规则优先拦截，通过后再进入 TMS 检测
- 互斥控制：TMS 与 AI 反骚扰只能二选一，开启 TMS 时自动关闭 AI 反骚扰
- 管理员命令：复用 /trust /untrust 命令（信任列表为同一套）
- 管理面板：新增 TMS 反骚扰配置开关

### 2.2 未来扩展

- 自定义审核策略（BizType 配置不同审核模板）
- 自定义词库集成
- 审核结果统计与分析

## 3. 核心设计原则

### 3.1 与 AI 反骚扰互斥

TMS 反骚扰与 AI 反骚扰是两种不同的检测手段，只能二选一：

| 维度 | AI 反骚扰 | TMS 反骚扰 |
|------|----------|------------|
| 检测方式 | LLM API（大语言模型） | 腾讯 TMS API（专业审核模型） |
| 响应速度 | 1-5 秒 | <100 毫秒 |
| 识别能力 | 语义理解，灵活 | 多维度覆盖，准确率高 |
| 成本 | LLM API 按 token 计费 | TMS 按条数计费（套餐包） |
| 互斥关系 | enable_ai_anti_harassment | enable_tencent_tms |

**互斥保证**：开启 TMS 时自动关闭 AI 反骚扰开关，反之亦然。两者不会同时生效。

### 3.2 信任列表共享

TMS 反骚扰与 AI 反骚扰共享同一套信任列表系统（user_trust 表）：

| 维度 | 说明 |
|------|------|
| 存储位置 | 同一张 user_trust 表 |
| 信任机制 | 同一套：当日连续通过 N 次晋升、每日重置 |
| 管理命令 | 同一套：/trust /untrust |
| 信任免检 | 同一逻辑：trusted 用户跳过检测 |

无论使用 AI 还是 TMS，信任列表的行为完全一致。切换检测方式时，信任数据不受影响。

### 3.3 信任每日重置

与 AI 反骚扰完全一致：
- 用户当天连续通过 N 次 TMS 检测 → 进入信任列表（trusted）
- 第二天首次发消息时 → 信任计数归零，状态回到 new
- 信任列表与黑白名单完全解耦

## 4. 详细需求

### 4.1 TMS 垃圾信息检测

| 需求项 | 说明 |
|--------|------|
| 检测时机 | 本地反骚扰检测通过后，消息转发前 |
| 检测对象 | 仅非信任用户的文本消息 |
| 检测范围 | 腾讯 TMS 覆盖的全部违规类型：色情、暴恐、违法、谩骂、广告、灌水等 |
| 检测方式 | 调用腾讯云 TextModeration API |
| 检测结果 | TMS 返回 Suggestion（Block/Review/Pass）+ Label + Score |
| 拦截判定 | Suggestion 为 Block 或 Review 且 Score >= 阈值 → 拦截 |
| 检测失败策略 | fail-open：TMS 不可用时放行消息，记录错误日志 |

**检测流程**：

```
用户消息 → 本地反骚扰检测 → 通过 → 黑名单检查 → 信任检查
                                                       ↓
                                                   trusted → 直接转发（跳过检测）
                                                   非trusted → TMS 检测
                                                               ↓
                                                         Block → 拉黑+通知管理员
                                                         Review(≥阈值) → 拉黑+通知管理员
                                                         Pass → 当日信任计数+1 → 转发
```

### 4.2 TMS 响应映射

TMS API 返回的 Label 和 Suggestion 需要映射为项目可理解的原因：

| TMS Label | 映射原因 | TMS Suggestion | 动作 |
|-----------|----------|----------------|------|
| Normal | - | Pass | 放行，信任计数+1 |
| Porn | 色情内容 | Block | 拦截 |
| Abuse | 辱骂内容 | Block | 拦截 |
| Ad | 广告内容 | Block | 拦截 |
| Illegal | 违法内容 | Block | 拦截 |
| Spam | 垃圾信息 | Block | 拦截 |
| Polity | 涉政内容 | Block | 拦截 |
| Terror | 暴恐内容 | Block | 拦截 |
| * | *(Label名称)* | Review | Score >= 阈值时拦截，否则放行 |

### 4.3 互斥控制

| 场景 | 动作 |
|------|------|
| 开启 TMS 反骚扰 | 自动关闭 AI 反骚扰（setConfig enable_ai_anti_harassment = false） |
| 开启 AI 反骚扰 | 自动关闭 TMS 反骚扰（setConfig enable_tencent_tms = false） |
| 同时开启 | 不允许，管理面板互斥校验 |
| 两者都关闭 | 消息仅做本地检测，正常转发 |

### 4.4 配置项

| 配置键 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `enable_tencent_tms` | boolean | `false` | TMS 反骚扰总开关 |
| `tencent_tms_review_block_threshold` | number | `60` | Review 结果拦截阈值（Score >= 此值时拦截） |
| `tencent_tms_trust_threshold` | number | `3` | 当日连续通过次数阈值（复用/共享信任表） |
| `tencent_tms_notify_auto_whitelist` | boolean | `true` | 自动加入信任列表时是否通知管理员 |

**环境变量**：

| 变量名 | 说明 | 示例 |
|--------|------|------|
| `TENCENT_SECRET_ID` | 腾讯云 API SecretId | `AKIDz8krbsJ5...` |
| `TENCENT_SECRET_KEY` | 腾讯云 API SecretKey | `Gu5t9geU...` |
| `TENCENT_TMS_REGION` | TMS API 地域 | `ap-guangzhou` |
| `TENCENT_TMS_TIMEOUT_MS` | TMS API 调用超时（毫秒） | `3000` |

**注意**：信任阈值配置 `ai_anti_harassment_trust_threshold` 与 `tencent_tms_trust_threshold` 分别设置，但实际读取信任阈值时，根据当前启用的检测方式选择对应的配置键。两者默认值相同（3）。

### 4.5 垃圾信息拦截动作

| 场景 | 动作 |
|------|------|
| TMS 检测为 Block | 设置 is_blocked=true（进入项目黑名单），记录垃圾原因到 user_info，调用 manageBlacklist()，发送拦截提示 |
| TMS 检测为 Review(>=阈值) | 同 Block 动作 |
| 管理员收到通知 | 在管理群组转发垃圾消息 + TMS 分析报告（含 Label、Suggestion、Score） |
| 用户收到提示 | "您的消息因包含垃圾信息已被过滤。如有疑问，请联系管理员。" |
| 信任列表影响 | TMS 检测为 Block 时，信任计数归零、状态转为 monitoring |

```
┌───────────────────┐
│ TMS 总开关？       │

      ▼          ▼
```

## 6. 非功能需求

### 6.1 性能要求

- TMS API 调用超时上限 3 秒，超时后 fail-open
- TMS 响应速度预期 <100ms，远优于 AI 反骚扰的 1-5s
- 信任用户不受 TMS 检测延迟影响
- 本地检测优先拦截，减少不必要的 TMS API 调用

### 6.2 成本控制

- 信任列表系统大幅减少 TMS API 调用量
- 仅文本消息触发 TMS 检测
- 仅非信任用户触发 TMS 检测
- 腾讯 TMS 套餐包计费（最低 2000 元/年/180 万条）

### 6.3 可靠性

- TMS API 不可用时 fail-open，不阻断正常消息
- TMS 检测错误不影响本地反骚扰功能
- 信任数据持久化到 D1（与 AI 反骚扰共享）

### 6.4 兼容性

- 与本地反骚扰检测并存，本地优先
- 与 AI 反骚扰互斥，不能同时启用
- 与项目黑白名单系统完全解耦
- 与现有管理面板兼容
- Cloudflare Workers 环境兼容（使用 fetch API + 手动 TC3-HMAC-SHA256 签名）

## 7. 验收标准

- [ ] TMS 检测为 Block 的消息被拦截，用户进入项目黑名单
- [ ] TMS 检测为 Review 且 Score >= 阈值 的消息被拦截
- [ ] TMS 检测为 Pass 的消息正常转发
- [ ] 信任用户当日消息直接转发，不调用 TMS API
- [ ] 当日连续通过 N 次 TMS 检测后进入信任列表
- [ ] 第二天首次发消息时信任计数归零，状态回到 new
- [ ] 开启 TMS 时自动关闭 AI 反骚扰
- [ ] 开启 AI 反骚扰时自动关闭 TMS
- [ ] 两者不能同时开启（管理面板互斥校验）
- [ ] /trust /untrust 命令在 TMS 模式下正常工作
- [ ] TMS API 不可用时消息正常放行（fail-open）
- [ ] TMS API 调用超时时消息正常放行
- [ ] 可通过配置开启/关闭 TMS 反骚扰功能
- [ ] 管理员收到垃圾消息分析报告（含 TMS Label/Suggestion/Score）
- [ ] 与本地反骚扰功能无冲突
- [ ] 信任列表与项目黑白名单互不影响（解耦验证）
