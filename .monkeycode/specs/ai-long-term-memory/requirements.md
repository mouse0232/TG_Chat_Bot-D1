# AI 长期记忆功能需求文档

## 1. 概述

### 1.1 背景

当前 Telegram 双向机器人的 AI 反骚扰检测功能依赖纯提示词 (prompt) 进行垃圾信息判断。这种方式存在以下局限：

- **漏判问题**：AI 可能无法识别某些新型垃圾信息模式，导致垃圾信息通过检测
- **误判问题**：AI 可能将正常消息误判为垃圾信息，影响用户体验
- **缺乏记忆能力**：每次检测都是独立的，AI 无法从历史判定中学习，同样的错误可能重复发生

### 1.2 目标

引入 Memobase 长期记忆系统，使 AI 能够：

- 记住历史判定结果和用户纠正反馈
- 在下次遇到类似消息时参考历史记忆，提高判定准确性
- 通过持续学习减少漏判和误判的发生率

### 1.3 核心价值

- **持续改进**：AI 判定能力随使用时间不断提升
- **用户反馈闭环**：管理员纠正后，AI 能够学习并避免同类错误
- **个性化检测**：针对不同用户的历史行为建立差异化检测策略

## 2. 需求规格 (EARS 格式)

### 2.1 功能需求

#### FR-001: 记忆存储
**WHEN** AI 完成一次垃圾信息检测判定时，**THEN** 系统应将以下信息存储到长期记忆：
- 用户 ID 和用户名
- 消息内容（截取前 512 字符）
- AI 判定结果（SPAM/CLEAN）
- 判定理由
- 时间戳

#### FR-002: 用户纠正反馈
**WHEN** 管理员发现 AI 误判或漏判并执行纠正操作时，**THEN** 系统应将纠正信息存储到长期记忆：
- 原始 AI 判定结果
- 管理员纠正后的正确结果
- 纠正理由/说明
- 原始消息内容

**WHEN** 管理员标记"漏判"（CLEAN → SPAM）时，**THEN** 系统应：
- 无论用户当前信任度如何，都将纠正记录直接存入 Memobase
- 立即触发"信任熔断"：清零该用户的信任度计数，移出信任名单
- 确保下次该用户发消息时重新进入 AI 检测流程，且 AI 能参考此次纠正记忆
- 执行拦截/拉黑操作（若未拦截）

**WHEN** 管理员标记"误判"（SPAM → CLEAN）时，**THEN** 系统应：
- 将纠正记录直接存入 Memobase
- **立即解除用户黑名单**（将该用户从 `is_blocked` 状态恢复为正常），使其后续消息能继续进入检测流程
- 发送误判纠正确认通知

#### FR-003: 信任熔断机制
**WHEN** 管理员确认某条消息为漏判的垃圾信息时，**THEN** 系统应：
- 立即将该用户的 AI 信任度清零（`consecutive_clean_count = 0`）
- 将该用户从信任名单中移除（`trust_status = 'monitoring'`）
- 无论该用户之前是否已达到免检信任阈值
- 信任熔断与记忆存储同步执行，互不依赖

**WHEN** 管理员标记"漏判"（CLEAN → SPAM）时，**THEN** 系统应：
- 无论用户当前信任度如何，都将纠正记录存入 Memobase
- 立即重置该用户的信任度（清零连续通过次数，移出信任名单）
- 确保下次该用户发消息时重新进入 AI 检测流程，并能参考此次纠正记忆

#### FR-003: 信任熔断机制
**WHEN** 管理员确认某条消息为漏判的垃圾信息时，**THEN** 系统应：
- 立即将该用户的 AI 信任度清零（`consecutive_clean_count = 0`）
- 将该用户从信任名单中移除（`trust_status = 'monitoring'`）
- 无论该用户之前是否已达到免检信任阈值

**WHEN** 管理员标记"漏判"（CLEAN → SPAM）时，**THEN** 系统应：
- 无论用户当前信任度如何，都将纠正记录存入 Memobase
- 立即重置该用户的信任度（清零连续通过次数，移出信任名单）
- 确保下次该用户发消息时重新进入 AI 检测流程，并能参考此次纠正记忆

#### FR-003: 记忆检索增强（全局模式泛化）
**WHEN** AI 需要对一条新消息进行垃圾信息检测时，**THEN** 系统应：
- 从 Memobase 检索 **全局纠正池** 的上下文（所有用户的历史纠正经验总结）
- 将全局规则注入 LLM 系统提示词，使 AI 能够应用 A 用户的纠正经验来判断 B 用户的消息
- 检索该特定用户的历史上下文（如有）作为补充参考

#### FR-004: 全局模式学习
**WHEN** 管理员对任何用户的消息执行纠正操作时，**THEN** 系统应：
- 将纠正记录（无论误判还是漏判）同时存储到 **全局纠正池**
- 利用 Memobase 的自动抽象能力，将具体的消息纠正转化为通用的判定规则
- 确保所有后续用户的检测都能受益于这些全局经验

#### FR-005: 记忆管理配置
**WHEN** 管理员在管理面板配置 AI 反骚扰功能时，**THEN** 系统应提供：
- 长期记忆功能总开关
- Memobase 服务地址配置
- Memobase API Key 配置
- 记忆保留时长配置

### 2.2 非功能需求

#### NFR-001: 性能要求
**WHEN** 系统调用 Memobase API 时，**THEN** 应满足：
- 记忆检索延迟不超过 500ms
- 记忆存储操作不应阻塞消息处理流程
- Memobase 服务不可用时，AI 检测应降级为无记忆模式继续工作 (fail-open)

#### NFR-002: 可靠性要求
**WHEN** Memobase 服务发生故障时，**THEN** 系统应：
- 记录错误日志但不中断消息处理
- 使用缓存的判定逻辑继续工作
- 在服务恢复后自动重试未成功的记忆操作

#### NFR-003: 数据安全要求
**WHEN** 存储用户消息到 Memobase 时，**THEN** 系统应：
- 仅存储必要的消息内容（不超过 512 字符）
- 不存储敏感个人信息（如电话号码、邮箱等）
- Memobase API Key 应通过环境变量安全存储

## 3. 用户故事

### 3.1 故事一：AI 误判纠正

**作为** 管理员  
**我希望** 能够在 AI 误判正常消息为垃圾信息时进行纠正  
**以便** AI 能够学习并避免对同类消息再次误判

**验收标准**：
1. 管理员收到 AI 误判通知后，可以点击"纠正"按钮
2. 纠正时需要选择正确的判定结果（CLEAN/SPAM）并可选填写理由
3. 纠正后，该用户的历史记忆中会新增一条纠正记录
4. 下次该用户发送类似消息时，AI 会参考纠正记录做出正确判定

### 3.2 故事二：AI 漏判发现

**作为** 管理员  
**我希望** 能够在 AI 漏判垃圾信息时手动标记  
**以便** AI 能够学习识别新型垃圾信息模式

**验收标准**：
1. 管理员可以在话题中标记某条消息为"垃圾信息"（无论用户是否已受信任）
2. 标记后，该消息的垃圾信息特征会被记录到 Memobase
3. 标记后，该用户的信任度立即清零（信任熔断）
4. 该用户下次发消息时重新进入 AI 检测，且 AI 会参考此次纠正记忆做出正确判定
5. 即使该用户之前已完成当日信任积累（如已发 5 条 CLEAN 消息），纠正仍然生效

### 3.3 故事三：记忆效果验证

**作为** 管理员  
**我希望** 能够查看 AI 长期记忆的学习效果  
**以便** 评估记忆功能是否有效并调整配置

**验收标准**：
1. 管理面板显示 AI 记忆学习统计（记忆条目数、纠正次数、判定准确率变化）
2. 可以查看特定用户的历史记忆摘要
3. 可以清除特定用户或全部记忆

## 4. 约束与依赖

### 4.1 技术约束

- 项目运行在 Cloudflare Workers 环境，需使用兼容的 HTTP 调用方式
- Memobase 需要独立部署（Docker 自托管或使用云服务）
- Node.js SDK `@memobase/memobase` 需验证与 Cloudflare Workers 的兼容性

### 4.2 外部依赖

- Memobase 后端服务（自托管或云端）
- LLM API（现有，需扩展提示词模板）

### 4.3 假设

- Memobase 服务稳定可用（SLA >= 99.9%）
- 用户消息内容不包含极端敏感信息
- 管理员会积极参与纠正反馈，形成有效的学习闭环

---

## 5. 成功指标

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| AI 判定准确率 | 提升 20%+ | 对比开启记忆功能前后的误判/漏判率 |
| 纠正反馈响应时间 | < 2 秒 | 管理员纠正操作到记忆存储完成的时间 |
| 记忆检索延迟 | < 500ms | 从 Memobase 获取上下文的时间 |
| 系统可用性 | 99.9% | Memobase 故障时降级不影响核心功能 |

---

## 6. 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| Memobase 服务不稳定 | AI 记忆功能不可用 | 中 | fail-open 策略，降级为无记忆模式 |
| 记忆质量差（噪声多） | AI 判定准确率下降 | 中 | 设置记忆筛选机制，仅存储高置信度记录 |
| API 调用延迟过高 | 消息处理超时 | 低 | 设置合理超时时间，异步处理记忆操作 |
| 存储成本超预期 | 运营费用增加 | 低 | 配置记忆保留时长，定期清理过期记忆 |
